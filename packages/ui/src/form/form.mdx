## Guidelines

`Field` is the labeled row that wraps a control: it renders the `<label>`, wires
it to the control with `htmlFor`/`id`, and shows an optional hint or error with
the correct `aria-describedby` association. Always place `Input` and `Textarea`
inside a `Field` so that wiring happens for you — a bare control loses the label
association.

The control is supplied through a render function, so any input type composes
in while `Field` still owns the id and description wiring:

```tsx
import { Field, Input, Textarea } from "@zudo-sg/ui";

<Field label="Email" required hint="We'll only use this to sign you in.">
  {(p) => <Input type="email" {...p} />}
</Field>

<Field label="Message">
  {(p) => <Textarea rows={6} {...p} />}
</Field>
```

## Hint vs error

- **Hint** — persistent guidance shown under the control (format, example, why
  you're asking). Neutral styling.
- **Error** — a validation message. When `error` is set it *replaces* the hint
  and turns the description red; render the control `invalid` at the same time
  so the border and focus ring match.

<Info>
`Field` auto-generates the control `id` when you don't pass `htmlFor`, using a
stable module counter (not a hook) so it renders identically on the server. Pass
`htmlFor` explicitly only when you need to reference the id elsewhere.
</Info>

## Do / Don't

- Do keep `invalid` on the control and `error` on the `Field` in sync — one
  without the other gives a mismatched signal (red text, neutral border, or the
  reverse).
- Do write hints as help, not restating the label ("As it appears on your
  passport", not "Enter your name").
- Don't use placeholder text as the label — it vanishes on input and is not a
  reliable accessible name.

<Warning>
An `error` message replaces the `hint`; you can't show both at once. Move any
always-relevant guidance into the label or a hint that only appears in the
valid state.
</Warning>

## Accessibility

- The label is associated via `for`/`id`, so clicking it focuses the control and
  screen readers announce the accessible name.
- Hint and error text are linked with `aria-describedby`, so the description is
  announced with the control.
- `required` renders a visible marker; pair it with the control's own `required`
  attribute so the requirement reaches the accessibility tree, not just the eye.
- An invalid control sets `aria-invalid`, so assistive tech announces the error
  state alongside the message.
